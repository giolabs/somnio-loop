# Comandos de Verificación Universales

Esta referencia es usada por `worker-dev`, `verifier`, `loop-self-healing` y `spec-refiner` para encontrar los comandos que verifican código en un proyecto dado.

## Orden de descubrimiento

Los comandos se resuelven en orden de prioridad. Los niveles inferiores solo se usan cuando los niveles superiores no dan resultado.

| Prioridad | Fuente | Cómo |
|---|---|---|
| 1 — Mayor | Task runner del proyecto | Leer `Makefile`, `justfile`, `Taskfile.yml`, `tasks.py`, `Rakefile`, aliases de `mix.exs`, `lefthook.yml` |
| 2 | Sección de scripts del manifest | `package.json` → `scripts`, `pyproject.toml` → tool scripts, `composer.json` → `scripts` |
| 3 — Fallback | Defaults idiomáticos por lenguaje | Tabla a continuación — solo cuando los pasos 1 y 2 no dan resultado |

## Comandos de fallback por stack

### TypeScript / JavaScript

| Acción | npm | pnpm | yarn | bun |
|---|---|---|---|---|
| Typecheck | `npx tsc --noEmit` | `pnpm exec tsc --noEmit` | `yarn tsc --noEmit` | `bun x tsc --noEmit` |
| Test | `npx jest` / `npx vitest run` | `pnpm test` | `yarn test` | `bun test` |
| Lint | `npx eslint .` | `pnpm lint` | `yarn lint` | `bun lint` |
| Build | `npx tsc` / `npx vite build` | `pnpm build` | `yarn build` | `bun run build` |

### Python

| Acción | Comando |
|---|---|
| Typecheck | `mypy <package>` / `pyright` |
| Test | `pytest` / `nox` / `tox` |
| Lint | `ruff check` / `flake8` / `pylint` |
| Format check | `ruff format --check .` / `black --check .` |

### Go

| Acción | Comando |

| Test | `go test ./...` |

## Regla fija

Al escribir el §8 del spec, siempre preferir los comandos del propio task runner o scripts del manifest del proyecto. La tabla de fallback es el último recurso para proyectos nuevos sin scripts configurados. Si incluso el fallback no aplica, marcar el §8 con `TBD — comandos de verificación faltantes; aclarar con el usuario antes de la implementación`.
